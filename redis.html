Base de datos en memoria, caché y broker de mensajes

¿Qué es Redis?

Redis es un almacén de estructuras de datos en memoria de código abierto utilizado como base de datos, caché y broker de mensajes.

  • Admite una amplia gama de estructuras de datos, como cadenas, hashes, listas, conjuntos y más.
  • Redis es conocido por su alto rendimiento, escalabilidad y simplicidad.
  • Redis puede proporcionar un rendimiento extremadamente rápido al almacenar toda su base de datos en memoria.

Características clave

Almacenamiento en memoria

Redis almacena los datos en RAM, lo que permite operaciones de lectura y escritura extremadamente rápidas.

Estructuras de datos

Redis proporciona un rico conjunto de estructuras de datos para manejar diversos casos de uso de manera eficiente.

Persistencia

Redis puede persistir los datos en disco, asegurando la durabilidad de los datos a través de reinicios del servidor.

Replicación

Redis admite la replicación maestro-esclavo y proporciona alta disponibilidad a través de Redis Sentinel.

Estructuras de Datos de Redis

Cadenas (Strings)

  • Las cadenas son el tipo de dato más básico en Redis.
  • Pueden almacenar cualquier tipo de dato: texto, números o datos binarios.
  • Redis proporciona comandos para manipular cadenas, con operaciones atómicas.
# Ejemplos de uso
SET usuario:1:nombre "Juan"
GET usuario:1:nombre
INCR contador
EXPIRE clave 60 # expira en 60s

Casos de uso:

Contadores, sesiones, almacenamiento de caché, semáforos, límite de ratio...

Listas (Lists)

  • Las listas en Redis son colecciones ordenadas de cadenas.
  • Se pueden utilizar para implementar colas, pilas o colas de mensajes.
  • Acceso por índice y operaciones en ambos extremos.
# Ejemplos de uso
LPUSH lista valor # añade al inicio
RPUSH lista valor # añade al final
LPOP lista # retira y devuelve del inicio
LRANGE lista 0 -1 # obtiene todos los elementos

Casos de uso:

Colas de mensajes, feeds sociales, historial de acciones recientes, colas de trabajo...

Conjuntos (Sets)

  • Los conjuntos son colecciones desordenadas de cadenas únicas.
  • Operaciones de conjuntos: unión, intersección, diferencia.
  • Útiles para eliminar duplicados y relaciones muchos a muchos.
# Ejemplos de uso
SADD amigos "Ana" "Luis" "Carlos"
SISMEMBER amigos "Ana" # verifica pertenencia
SCARD amigos # cuenta elementos
SINTER set1 set2 # intersección de conjuntos

Casos de uso:

Etiquetas, seguimiento de relaciones, IP únicas, eliminación de duplicados...

Hashes

  • Los hashes son pares clave-valor dentro de una sola clave de Redis.
  • Perfectos para representar objetos y documentos.
  • Más eficientes que almacenar cada campo como una clave separada.
# Ejemplos de uso
HSET usuario:1 nombre "Juan" edad 28 ciudad "Madrid"
HGET usuario:1 nombre
HINCRBY usuario:1 visitas 1
HGETALL usuario:1 # obtiene todos los campos y valores

Casos de uso:

Perfiles de usuario, configuraciones, datos de sesión, carrito de compras...

Conjuntos Ordenados (Sorted Sets)

  • Conjuntos donde cada elemento tiene una puntuación (score).
  • Los elementos se ordenan automáticamente por su puntuación.
  • Consultas eficientes por rango de puntuación o posición.
# Ejemplos de uso
ZADD puntuaciones 90 "Ana" 85 "Luis" 95 "Carlos"
ZRANGE puntuaciones 0 -1 # todos ordenados
ZREVRANGE puntuaciones 0 2 # 3 mejores
ZRANGEBYSCORE puntuaciones 80 100 # por rango

Casos de uso:

Tablas de clasificación, accesos recientes, analíticas por tiempo, priorización...

Casos de Uso de Redis

Caché

Redis es ampliamente utilizado como capa de caché para mejorar el rendimiento de aplicaciones, reduciendo la latencia de acceso a datos y la carga en la base de datos principal.

Beneficios:

  • Velocidad de respuesta: acceso en memoria <1ms
  • Reducción de carga en bases de datos
  • Políticas de expiración automáticas
  • Algoritmos LRU/LFU para gestión de memoria

Gestión de Sesiones

Redis es ideal para almacenar datos de sesión de usuario en aplicaciones web, permitiendo acceso rápido y consistente a través de múltiples servidores.

Ventajas:

  • Centralización de datos de sesión
  • Escalabilidad horizontal entre servidores
  • Tiempos de expiración configurables
  • Alta disponibilidad para sesiones críticas

Análisis en Tiempo Real

La naturaleza en memoria de Redis lo hace ideal para analíticas y paneles en tiempo real, permitiendo procesar grandes volúmenes de datos con respuesta inmediata.

Aplicaciones:

  • Contadores de eventos en tiempo real
  • Tablas de clasificación y trending
  • Seguimiento de comportamiento de usuario
  • Analítica de tráfico y métricas

Mensajería Pub/Sub

Redis proporciona un sistema de mensajería Publicación/Suscripción ligero y de alto rendimiento para comunicación entre componentes de aplicaciones.

Características:

  • Comunicación en tiempo real
  • Modelo sencillo de publicadores y suscriptores
  • Distribución de mensajes a múltiples consumidores
  • Alto rendimiento y baja latencia

Persistencia en Redis

A pesar de ser una base de datos en memoria, Redis ofrece mecanismos de persistencia para garantizar la durabilidad de los datos.

RDB (Redis Database)

Persistencia mediante snapshots puntuales de la base de datos completa.

  • Crea instantáneas en momentos específicos (configurables)
  • Los archivos RDB son compactos y optimizados
  • Ideal para backups y recuperación ante desastres
  • Menor sobrecarga de I/O durante operación normal
# Configuración en redis.conf
save 900 1 # cada 15min si ≥1 cambio
save 300 10 # cada 5min si ≥10 cambios
save 60 10000 # cada 1min si ≥10000 cambios

# Comando manual
BGSAVE # inicia snapshot en background

AOF (Append-Only File)

Persistencia mediante registro de todas las operaciones de escritura.

  • Registra cada operación de escritura en un archivo
  • Mayor durabilidad (menor pérdida de datos)
  • Diferentes niveles de sincronización (fsync)
  • La compactación automática evita crecimiento excesivo
# Configuración en redis.conf
appendonly yes
appendfsync everysec # opciones: always, everysec, no

# Reescritura del AOF
BGREWRITEAOF # compacta el AOF en background

Comparación de estrategias de persistencia

Característica RDB AOF
Durabilidad Menor (intervalos entre snapshots) Mayor (registro continuo)
Tamaño de archivo Compacto Más grande
Impacto en rendimiento Menor en general, picos al hacer snapshot Constante pero ligero, mayor con fsync=always
Tiempo de recuperación Rápido (carga directa) Más lento (replay de operaciones)
Resistencia a corrupción Vulnerable (formato binario) Mayor (operaciones secuenciales)

Recomendación: Usar ambas estrategias combinadas para máxima seguridad y rendimiento

Replicación y Alta Disponibilidad

Replicación Maestro-Réplica

Redis implementa un sistema de replicación asíncrona donde los nodos réplica mantienen una copia exacta de los datos del nodo maestro.

Características

  • Configuración simple y robusta
  • Escalabilidad horizontal de lecturas
  • Sincronización automática después de desconexión
  • Transferencia parcial para reconexiones rápidas
# En nodo réplica:
replicaof 192.168.1.10 6379

# O en tiempo de ejecución:
REPLICAOF 192.168.1.10 6379
REPLICAOF NO ONE # promueve a maestro

Flujo de replicación:

  1. Réplica conecta al maestro
  2. Maestro inicia sincronización completa (o parcial)
  3. Transferencia de RDB del estado actual
  4. Réplica carga datos y actualiza
  5. Propagación continua de comandos

Redis Sentinel

Redis Sentinel proporciona alta disponibilidad para Redis a través de monitorización automatizada, notificación y conmutación por error automática.

Funciones principales

  • Monitorización: verifica que maestro y réplicas funcionan correctamente
  • Notificación: alerta a administradores o sistemas sobre problemas
  • Failover: promoción automática de réplica a maestro cuando falla el maestro
  • Configuración cliente: actúa como servicio de descubrimiento
# sentinel.conf
sentinel monitor mymaster 192.168.1.10 6379 2
sentinel down-after-milliseconds mymaster 5000
sentinel failover-timeout mymaster 60000
sentinel parallel-syncs mymaster 1

Funcionamiento del quórum:

Sentinel utiliza algoritmos de consenso para detectar fallos y evitar falsos positivos. El valor del quórum (2 en el ejemplo) determina cuántos sentinels deben estar de acuerdo para iniciar una conmutación por error.

Recomendación: al menos 3 instancias de Sentinel en diferentes hosts para tolerancia a fallos.

Redis Cluster

Distribución de datos horizontal

Redis Cluster permite distribuir automáticamente los datos entre múltiples nodos Redis, ofreciendo escalabilidad horizontal y alta disponibilidad.

Características clave

  • Distribución automática en 16384 slots de hash
  • Conmutación por error automática entre nodos
  • Comunicación entre nodos mediante gossip protocol
  • Soporte para múltiples conexiones simultáneas
  • Escalabilidad hasta cientos o miles de nodos

Cómo funciona el sharding:

Las claves se asignan a un slot de hash mediante CRC16(key) % 16384. Cada nodo maestro es responsable de un subconjunto de estos slots. Los nodos replican entre sí información sobre la asignación de slots.

Configuración de Cluster

# redis.conf (para cada nodo)
port 7000
cluster-enabled yes
cluster-config-file nodes-7000.conf
cluster-node-timeout 5000
appendonly yes

Para crear el cluster (en Redis 5.0+):

redis-cli --cluster create 127.0.0.1:7000 127.0.0.1:7001 \
127.0.0.1:7002 127.0.0.1:7003 127.0.0.1:7004 127.0.0.1:7005 \
--cluster-replicas 1

Buenas prácticas:

  • • Mínimo 3 nodos maestros para tolerancia a fallos
  • • Un nodo réplica por cada maestro
  • • Nodos en diferentes hosts/racks para máxima disponibilidad
  • • Monitorizar el ancho de banda de red entre nodos

Limitaciones y consideraciones

  • Transacciones Multi-Key

    Las operaciones multi-clave solo funcionan cuando todas las claves pertenecen al mismo slot de hash

  • Complejidad del sistema

    Mayor complejidad operativa y de mantenimiento que Redis Standalone

  • Tamaño de red

    El tráfico de gossip puede aumentar significativamente con muchos nodos

  • Redistribución de datos

    Añadir/quitar nodos requiere redistribución de slots que consume recursos

Redis Cluster es ideal cuando necesitas:

• Escalar más allá de la capacidad de un solo nodo Redis
• Alta disponibilidad sin intervención manual
• Conjunto de datos distribuido con tolerancia a fallos

Seguridad en Redis

Autenticación

Redis proporciona un mecanismo simple de autenticación basado en contraseña para proteger el acceso a los datos.

# Configurar contraseña en redis.conf
requirepass "contraseña_fuerte_y_compleja"

# O en tiempo de ejecución
CONFIG SET requirepass "contraseña_fuerte_y_compleja"

# Autenticarse como cliente
AUTH "contraseña_fuerte_y_compleja"

Buenas prácticas:

  • Usar contraseñas largas y complejas (>20 caracteres)
  • Rotar contraseñas regularmente
  • Usar variables de entorno en producción
  • Combinar con restricciones de red

Control de Acceso (ACL)

A partir de Redis 6.0, se introdujo un sistema de Control de Acceso basado en usuarios (ACL) que permite un control más granular.

# Crear un usuario con acceso restringido
ACL SETUSER alice on >secret_pwd ~objects:* +@read +@hash -@all

# Explicación:
# "on" - usuario activo
# ">secret_pwd" - contraseña
# "~objects:*" - acceso solo a claves que empiezan con "objects:"
# "+@read" - permite comandos de lectura
# "+@hash" - permite comandos para trabajar con hashes
# "-@all" - deniega todos los demás comandos

Ventajas del ACL:

  • Principio de mínimo privilegio
  • Separación de responsabilidades
  • Restricción por patrón de claves
  • Auditoría y control detallado

Cifrado en tránsito (TLS)

Redis 6.0+ soporta TLS nativo para cifrar la comunicación entre clientes y servidores, así como entre réplicas.

Configuración del servidor

# En redis.conf
tls-port 6379
tls-cert-file /path/to/redis.crt
tls-key-file /path/to/redis.key
tls-ca-cert-file /path/to/ca.crt
tls-auth-clients yes
tls-replication yes

Conexión del cliente

# Usando redis-cli
redis-cli --tls \
--cert /path/to/client.crt \
--key /path/to/client.key \
--cacert /path/to/ca.crt

Mejores prácticas de seguridad

Red y aislamiento

  • Usa bind para limitar interfaces de red
  • Implementa Redis en redes privadas/VPC
  • Configura firewalls y grupos de seguridad
  • Usa proxies para proteger acceso directo

Configuración protegida

  • Deshabilita comandos peligrosos en producción
  • Limita memoria máxima (maxmemory)
  • Establece límites de clientes (maxclients)
  • Monitoriza y audita regularmente

⚠️ Advertencia:

Nunca exponga instancias de Redis directamente a Internet o redes públicas sin la debida protección. Redis fue diseñado para entornos de confianza y debería utilizarse siempre detrás de firewalls y con autenticación adecuada.
